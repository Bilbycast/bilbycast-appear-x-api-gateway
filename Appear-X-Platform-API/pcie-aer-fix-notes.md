# MS-02 Ultra — PCIe AER error flood: fix notes

**Date applied:** 2026-05-01
**Hostname:** ms02-ultra-1
**System:** Minisforum MS-02 Ultra (board: Meigao/MorefIne ARWSA), Intel Arrow Lake-HX, 800-series PCH
**OS at time of fix:** Ubuntu 26.04 LTS, kernel `7.0.0-15-generic`
**BIOS at time of fix:** AMI 1.04, dated 2026-04-21

---

## Symptom

Login screen and `journalctl` flooded with PCIe Advanced Error Reporting (AER) messages — roughly **12,000+ per boot**, growing continuously. Examples:

```
pcieport 0000:80:1b.4: AER: Uncorrectable (Non-Fatal) error message received from 0000:80:1b.4
pcieport 0000:80:1b.4: PCIe Bus Error: severity=Uncorrectable (Non-Fatal), type=Transaction Layer
pcieport 0000:80:1b.4:    [21] ACSViol                (First)
pcieport 0000:80:1b.4:    [20] UnsupReq               (First)
thunderbolt 0000:84:00.0: AER: can't recover (no error_detected callback)
xhci_hcd   0000:97:00.0: AER: can't recover (no error_detected callback)
pcieport 0000:80:1b.4: AER: device recovery failed
```

## Root cause

The errors all originated at **PCIe Root Port #21 (`0000:80:1b.4`, `[8086:7f44]`)** in the 800-series PCH. The only thing downstream of this port is the **Intel JHL9580 "Barlow Ridge" Thunderbolt 5 80/120G** controller (`0000:82:00.0` → bridge → `84:00.0` TB NHI + `97:00.0` xHCI).

The TB5 controller emits spurious **ACSViol** (Access Control Services Violation) and **UnsupReq** (Unsupported Request) TLPs even with no external Thunderbolt device attached. The kernel logs every one. "Recovery" fails because the `thunderbolt` and `xhci_hcd` drivers don't implement AER `error_detected` callbacks — but there is nothing to recover from; the errors are **non-fatal** and benign.

This is a known platform-level issue with the **Arrow Lake + 800-PCH + JHL9580 Barlow Ridge** combination. BIOS 1.04 (the latest available at fix time) does not address it. No Thunderbolt firmware update was offered for this board via `fwupdmgr`.

The 25 GbE NICs (Intel E810-XXV at `01:00.0` / `01:00.1`), NVMe, Realtek 10G, Intel I226 2.5G, and Wi-Fi 7 are on **different PCIe root ports** and are entirely unrelated to the noisy port.

## Why a surgical setpci mask did not work

The first attempt was a `systemd` unit running `setpci` to mask UnsupReq + ACSViol bits in the AER Uncorrectable Error Mask register on `0000:80:1b.4`. **It failed with `pcilib: sysfs_write: write failed: Operation not permitted`.**

Reason: when the OS owns AER (visible in dmesg as `acpi PNP0A08:00: _OSC: OS now controls [PCIeHotplug SHPCHotplug PME AER PCIeCapability LTR DPC]`), modern Linux kernels block userspace writes to the AER capability through `/sys/bus/pci/devices/.../config`, even as root. This is intentional kernel hardening; there is no userspace knob to write that register. The broken systemd unit was removed.

A more surgical fix would require a small custom kernel module (DKMS) calling `pci_disable_pcie_error_reporting()` on the affected device — not pursued here because the broad cmdline option is sufficient on this consumer hardware.

## What was actually changed

A single kernel command-line flag added via GRUB:

| File | Change |
|---|---|
| `/etc/default/grub` | `GRUB_CMDLINE_LINUX_DEFAULT=""` → `GRUB_CMDLINE_LINUX_DEFAULT="pci=noaer"` |
| `/etc/default/grub.bak` | Created as backup of the original |
| `/boot/grub/grub.cfg` | Regenerated by `sudo update-grub` |

Effect of `pci=noaer`: kernel does not initialize the AER service for any PCIe device. No AER capability claim, no AER interrupts, no AER log messages, no AER-mediated recovery callbacks.

## What this does **not** affect

PCIe data paths, link operation, hot-plug (`pciehp`), MSI/MSI-X, ASPM, IOMMU, suspend/resume, and every device driver continue to work normally. Real failures still surface through other channels: block I/O errors, network link-state changes, SMART, smartctl, dmesg link messages.

Confirmed working post-reboot:
- All NICs enumerated (`eno1np0`, `eno2np1` = E810-XXV 25G; `eno3` = 2.5G; `eno4` = Realtek 10G; `wlo5` = Wi-Fi 7)
- Thunderbolt 5 controllers visible (`/sys/bus/thunderbolt/devices/0-0`, `1-0`, `domain0`, `domain1`)
- Samsung 990 PRO 2 TB NVMe present
- AER messages this boot: **0** (was ~12,000 per boot before)

## What you lose

The kernel no longer emits AER-formatted PCIe error reports on any device. Tools that consume AER (`rasdaemon`, enterprise RAS dashboards) will have nothing to log. Functional impact for desktop / lab use: none.

## Verification (run any time)

```bash
echo "--- cmdline (should contain pci=noaer) ---"
cat /proc/cmdline
echo "--- AER count this boot (expect 0) ---"
journalctl -kb | grep -c AER
echo "--- thunderbolt ---"
ls /sys/bus/thunderbolt/devices/
echo "--- NICs ---"
ip -br link
echo "--- NVMe ---"
lsblk -d -o NAME,SIZE,MODEL | grep -i nvme
```

## Rollback (re-enable AER)

If a future BIOS update or Barlow Ridge firmware fixes the underlying issue, or if you start running RAS tooling that needs AER:

```bash
# Quickest: restore backup
sudo cp /etc/default/grub.bak /etc/default/grub
sudo update-grub
sudo reboot
```

Or edit `/etc/default/grub` directly and set `GRUB_CMDLINE_LINUX_DEFAULT=""` (or whatever else you want there), then `sudo update-grub` and reboot.

## When to revisit

- After a **BIOS update** for the MS-02 Ultra (current is 1.04 / 2026-04-21). Check vendor page and try removing `pci=noaer` to see if the spam is gone.
- After a major **kernel update** (a future kernel may add a per-device AER quirk for JHL9580).
- After running **`fwupdmgr update`** if it ever offers a JHL9580 / Barlow Ridge Thunderbolt controller firmware update.
- If you ever need detailed PCIe RAS observability (e.g. running `rasdaemon`).
